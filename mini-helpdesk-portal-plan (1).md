# Mini Helpdesk Portal — Technical Design Specification
### Laravel 13 Fullstack | Spec-Driven Development Plan

---

## 0. How to use this document

Each phase below is a self-contained spec: implement, verify against its acceptance criteria, commit, then move to the next. The database and authorization phases are load-bearing for everything after — don't parallelize past them.

---

## 1. Architecture Decisions

| Decision | Choice | Why |
|---|---|---|
| Framework | Laravel 13 | Given |
| PHP version | 8.4+ | Laravel 13 minimum requirement |
| Frontend approach | **Blade + Livewire 4 + Alpine.js** | Fullstack in one language, no separate API/SPA layer, reactive UI (filters, live tables) without a heavy JS build. Livewire 4 (not 3) — it's the version compatible with Laravel 13 and is the current stable release |
| Styling | Tailwind CSS via Vite | Ships with Laravel 13 by default |
| Auth scaffold | **Official `laravel/livewire-starter-kit`** (not Breeze) | Breeze/Jetstream were dropped from the `laravel new` installer as of Laravel 12 and replaced by first-party starter kits; the current Livewire kit is built on Livewire 4 + Tailwind + Flux UI + Laravel Fortify for auth (login, password reset, email verification). Public `/register` is disabled via Fortify's feature toggle — user creation is Admin-only |
| UI components | Flux UI (free tier) | Ships with the official starter kit — buttons, dropdowns, modals, form inputs, already accessible and dark-mode aware, so CRUD screens don't need hand-rolled components |
| Authorization | Laravel **Policies + Gates**, role stored as a backed PHP enum | Static 3-role system doesn't need a permissions package like Spatie |
| Enums | PHP 8.1+ backed enums for `Role`, `TicketStatus`, `Priority`, `ClientStatus` | Type-safe, IDE-autocompletable, invalid values are impossible instead of just discouraged |
| Validation | Form Requests (`StoreTicketRequest`, `UpdateTicketRequest`, `StoreClientRequest`, `StoreUserRequest`, etc.) | Reusable, testable independent of the Livewire component, keeps components focused on UI state |
| Repositories | **Not used** | With Eloquent + Form Requests + a couple of Service classes, a repository layer is ceremony for this scope, not abstraction that earns its keep |
| Query centralization | `TicketQueryService` as the single place ticket-scoping logic lives, consumed by `DashboardMetricsService` and `MonthlyReportService` | Avoids the same "which tickets can this user see" logic being reimplemented three times |
| Charts | Chart.js rendered inside Blade/Livewire | Lightweight, no extra backend |
| Database | MySQL 8 (SQLite for local/dev) | Standard Laravel default |
| Testing | Pest | Feature tests per role, per module |
| Containerization | Docker Compose (php, nginx, mysql, mailpit) | `docker compose up` reproducibility |
| PDF export (optional) | `barryvdh/laravel-dompdf` only if implemented; otherwise `window.print()` + print CSS | Keep scope small |

**Not used:** Inertia+React/Vue, Sanctum API tokens, queues/websockets, repository pattern — unnecessary complexity for this scope.

## 4. Roles & Authorization Matrix

Laravel Policies (`ClientPolicy`, `UserPolicy`, `TicketPolicy`, `TicketCommentPolicy`), registered centrally. A `Ticket::scopeVisibleTo(Builder $query, User $user)` query scope encapsulates the "which tickets can this user see" logic once, so it's called the same way everywhere instead of re-derived per controller/component:

```php
// Ticket model
public function scopeVisibleTo(Builder $query, User $user): Builder
{
    return match ($user->role) {
        Role::Admin => $query,
        Role::Engineer => $query->where('assigned_engineer_id', $user->id),
        Role::Client => $query->where('client_id', $user->client_id),
    };
}

// usage anywhere
Ticket::visibleTo(auth()->user())->get();
```

`TicketPolicy::view()` should call the same underlying check, so the query scope and the single-model authorization check can never drift apart.

| Action | Admin | Engineer | Client |
|---|---|---|---|
| View all clients | ✅ | ❌ | ❌ |
| Create/edit client | ✅ | ❌ | ❌ |
| View all users | ✅ | ❌ | ❌ |
| Create/edit user | ✅ | ❌ | ❌ |
| View tickets | ✅ all | assigned only | own client only |
| Create ticket | ✅ any client | ❌ | ✅ own client only |
| Assign engineer | ✅ | ❌ (self-assign only if enabled) | ❌ |
| Change priority | ✅ | ❌ | ❌ locked after creation |
| Change status | ✅ | ✅ own assigned | ❌ |
| Add public comment | ✅ | ✅ | ✅ own tickets only |
| Add internal comment | ✅ | ✅ | ❌ |
| View monthly report | ✅ any client | out of scope | ✅ own client only |

## 6. Folder Structure

```

  Enums/ (Role, TicketStatus, Priority, ClientStatus)
  Models/ (User, Client, Ticket, TicketComment, TicketStatusHistory, MonthlyReportRemark)
  Policies/ (ClientPolicy, UserPolicy, TicketPolicy)
  Http/Requests/ (StoreTicketRequest, UpdateTicketRequest, StoreClientRequest, StoreUserRequest, ...)
  Observers/
    TicketObserver.php        <- assigns ticket_number on creating(), writes status history on updating()
  Services/
    TicketNumberService.php
    TicketQueryService.php    <- central scoping/filtering, used by dashboards + reports
    DashboardMetricsService.php
    MonthlyReportService.php
  Livewire/
    Dashboard/ (AdminDashboard, EngineerDashboard, ClientDashboard)
    Clients/ (ClientTable, ClientForm)
    Users/ (UserTable, UserForm)
    Tickets/ (TicketTable, TicketForm, TicketDetail, CommentThread)
    Reports/ (MonthlyReport)
database/
  migrations/
  factories/
  seeders/ (DatabaseSeeder, UserSeeder, ClientSeeder, TicketSeeder)
resources/views/livewire/...
tests/Feature/ (per role, per module)
docker/
  docker-compose.yml
  php/Dockerfile
  nginx/default.conf
docs/
  SPEC.md    <- this document, referenced during implementation
  ERD.png (or .mermaid source)
```

Using an **Observer** instead of a model `booted()` hook for ticket-number generation and status-history logging keeps `Ticket.php` a thin data model — creation-time side effects live in `TicketObserver`, not scattered across lifecycle hooks inside the model class itself.

## 7. Activity Logging

Minimal, no external package required — log key domain events via Laravel's built-in logger with a structured channel, or a lightweight `activity_logs` table if a visible audit trail in the UI is wanted:

- Ticket created
- Ticket assigned / reassigned
- Status changed (already captured richly in `ticket_status_histories`, which doubles as the audit source for tickets)
- User created / deactivated
- Client created / deactivated

If kept to Laravel's log driver, this is a `TicketObserver`/`UserObserver` one-liner per event (`Log::info('ticket.assigned', [...])`) — cheap to add, useful in review, not a project in itself.

## 8. Development Phases

### Phase 0 — Project Bootstrap
- `laravel new mini-helpdesk-portal` and select the **Livewire** starter kit when prompted (or `composer create-project laravel/livewire-starter-kit` if not using the interactive installer). This scaffolds Livewire 4, Tailwind, Flux UI, and Fortify-based auth out of the box — no separate Breeze install step.
- Add Chart.js (Flux Pro has built-in charts, but Chart.js keeps the free-tier build simple and dependency-light for this project).
- In `config/fortify.php`, disable the `registration` feature so `/register` is unreachable — user creation is Admin-only via `/users`.
- Docker Compose scaffold (php, nginx, mysql, mailpit).
- **Acceptance:** `docker compose up` serves the starter kit's login page; `/register` returns 404/redirect, not a working form.

### Phase 1 — Database, Enums & Models
- Migrations per Section 3, including all listed indexes.
- Enums per Section 2, cast on models.
- Models with relationships and the `scopeVisibleTo()` query scope on `Ticket`.
- `TicketObserver` for ticket-number generation + status-history writes.
- Factories + seeders producing the PRD seed data.
- **Acceptance:** `php artisan migrate:fresh --seed` runs clean; `Ticket::visibleTo($engineer)->get()` in `tinker` returns only that engineer's assigned tickets.

### Phase 2 — Auth & Role Routing
- Remove public `/register`; user creation is Admin-only via `/users`.
- Login redirect by `role` enum to the correct dashboard.
- Policies registered; `role` middleware on Admin-only route groups.
- **Acceptance:** each seeded user lands on the correct dashboard; non-admin hitting `/clients` gets 403.

### Phase 3 — Client Management (Admin only)
- `ClientTable` (list, search, status toggle) + `ClientForm`, validated via `StoreClientRequest` / `UpdateClientRequest`.
- **Acceptance:** create/edit persist correctly; deactivate flips `status` without deleting the row; non-admin blocked at the route.

### Phase 4 — User Management (Admin only)
- `UserTable` + `UserForm`, validated via `StoreUserRequest` / `UpdateUserRequest`.
- Conditional `client_id` field, required only when `role === Role::Client`.
- Password hashed on save, never rendered back.
- **Acceptance:** creating a Client-role user without `client_id` fails validation server-side (test via direct POST, not just UI); email uniqueness enforced.

### Phase 5 — Ticket Module
- `TicketForm` validated via `StoreTicketRequest` (client, title, description, priority) / `UpdateTicketRequest` (status, assignment, priority — Admin only).
- `TicketTable` using `TicketQueryService` for role-scoped listing + filters.
- `TicketDetail` with role-conditional action panel.
- `CommentThread` with public/internal toggle (admin/engineer only).
- Status changes routed through `TicketObserver` to populate `ticket_status_histories` (including `notes`), and set `resolved_at` on transition to Resolved.
- **Acceptance:** Pest feature tests confirm each role's visible ticket set matches Section 4's matrix; a Client-role direct POST attempting to change `priority` is rejected server-side regardless of UI state.

### Phase 6 — Dashboards
- `TicketQueryService` provides scoped base queries; `DashboardMetricsService` builds on top of it for `adminSummary()`, `engineerSummary()`, `clientSummary()` — real aggregate queries, nothing hardcoded.
- Three Livewire dashboards: cards + Chart.js status breakdown + latest-5 table.
- **Acceptance:** changing a ticket's status via `tinker` changes dashboard numbers on reload; numbers cross-checked against a manual `GROUP BY status` query.

### Phase 7 — Monthly Report
- `MonthlyReportService`, built on `TicketQueryService`, returns totals/status/priority breakdowns, ticket list, and saved remark for `(client_id, month, year)`.
- `MonthlyReport` Livewire component: selector (client fixed for Client role), report body, remarks field (Admin input per PRD), print view/stylesheet.
- **Acceptance:** empty month renders a zeroed report, not an error; print view is clean (no nav/sidebar) on one page.

### Phase 8 — Testing, Hardening, Docs
- Pest feature tests per role covering allowed/denied access on every module.
- Confirm no raw password appears in any view/log; confirm no delete route exists for tickets/clients.
- `README.md` covering: architecture overview, ERD, folder structure, setup (including `docker compose up` path), seed credentials, role summary, known limitations, and possible future improvements (not just install steps).
- **Acceptance:** `php artisan test` green; a reviewer can clone, run `docker compose up`, migrate/seed, and log in as each seeded account.

## 10. Explicit Non-Goals

- No email/SMS notifications, no queues/broadcasting.
- No API layer — server-rendered only.
- No repository pattern — Eloquent + Form Requests + Services is sufficient at this scope.
- No permanent deletion of tickets or clients.
- PDF export optional; browser print satisfies the requirement.
